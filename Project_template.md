### 2. Анализ архитектуры монолитного приложения
- Язык программирования: Go
- СУБД: PostgreSQL
- Архитектура: Монолитная — все компоненты (API, бизнес-логика, доступ к данным, интеграция с устройствами) объединены в одно приложение.
- Взаимодействие: Синхронное, блокирующее. Сервер инициирует запрос к датчикам, ожидает ответа.
- Масштабируемость: Ограничена. Масштабирование возможно только "целиком", что неэффективно при росте нагрузки на отдельные функции.
- Развёртывание: Требует полной остановки и перезапуска приложения, что приводит к простою.
- Гибкость: Низкая. Добавление новых типов устройств или функций требует изменения кода монолита и повторного деплоя.
- Самообслуживание: Не поддерживается. Подключение устройств возможно только через специалиста.
  
### 3. Определение доменов и границы контекстов
На основе анализа функциональности и требований выделены следующие домены (Bounded Contexts):

- **Device Management (Управление устройствами)** — регистрация, статус и конфигурация устройств.
- **Thermostat (Отопление)** — управление отоплением и мониторинг температуры.
- **Lighting (Освещение)** — управление светом в помещениях.
- **Surveillance (Наблюдение)** — интеграция с камерами, просмотр видео.
- **Access Control (Контроль доступа)** — управление воротами, дверями, замками.
- **User & Auth (Пользователи и авторизация)** — управление аккаунтами, вход, права доступа.
- **Automation (Автоматизация)** — создание и выполнение сценариев.
- **Telemetry (Телеметрия)** — сбор и хранение данных с устройств.

Каждый домен будет реализован как отдельный микросервис с чётко определённым API и собственной базой данных.

### 4. Проблемы монолитного решения
- **Низкая масштабируемость**: невозможно масштабировать отдельные компоненты (например, сбор телеметрии), приходится масштабировать всё приложение целиком.
- **Высокий риск при деплое**: любое изменение требует перезапуска всего сервиса, что может привести к простою.
- **Ограниченная гибкость**: добавление новых типов устройств (освещение, камеры) требует изменения кода монолита, что замедляет разработку.
- **Отсутствие самообслуживания**: пользователи не могут подключать устройства самостоятельно — требуется выезд специалиста.
- **Синхронная архитектура**: блокирующие вызовы могут привести к зависанию системы при проблемах с одним устройством.
- **Высокая связанность**: изменение в одной части системы может повлиять на другие, что усложняет тестирование и поддержку.

## Задание 4. Создание и документирование API

### 1. Тип API
Для взаимодействия микросервисов используется **REST over HTTP/JSON**.

**Обоснование**:
- REST — стандарт для SaaS-приложений и микросервисов.
- Хорошо поддерживается в экосистеме Java (Spring Boot).
- Прост в разработке, тестировании и документировании.
- Подходит для синхронного взаимодействия между фронтендом и бэкендом.
- Легко интегрируется с OpenAPI (Swagger) для документации.

## Задание 5. Работа с docker и docker-compose

Приложение `temperature-api` реализовано на **Spring Boot (Java 17)** и упаковано в Docker.

- Сервис доступен на порту `8081`
- Используется `docker-compose.yml` для запуска:
    - `temperature-api` — микросервис на Spring Boot
    - `smart_home_db` — PostgreSQL с инициализацией через `init.sql`
- При каждом запросе к `/temperature` возвращается **случайное значение температуры**
- Логика обработки параметров `location` и `sensorId` соответствует заданию
